## Problem Statement

Write an annotated linker script (in a comment-based simulation) that defines the memory layout for a typical ARM Cortex-M MCU. Include flash and RAM regions, place the vector table, text, data, BSS, heap, and stack. Explain each section's purpose and how the startup code uses the linker symbols.

## Theory and Concepts

- The linker script defines the memory map: FLASH (rx) for code and constants, RAM (rwx) for data and stack.
- Sections: `.isr_vector` at the start of flash, `.text` for code, `.rodata` for constants, `.data` for initialised variables (VMA in RAM, LMA in flash), `.bss` for zero-initialised variables, `.heap` and `.stack` for runtime memory.
- Linker symbols (`_sdata`, `_edata`, `_sidata`, `_sbss`, `_ebss`, `_estack`, `_sheap`, `_eheap`) are generated by the linker and used by the startup code.
- `ENTRY(Reset_Handler)` defines the entry point.
- `PROVIDE()` defines a symbol with a default value that can be overridden by the application.
- The `NOCROSSREFS` or `NOCROSSREFS_TO` directives check section placement is correct.

## Real World Application

The linker script is one of the most important files in an embedded project — it determines where code and data reside in memory. Understanding it is essential for bootloader development, implementing firmware update mechanisms (dual-bank), creating custom memory layouts, and debugging hard-to-find memory corruption issues.
